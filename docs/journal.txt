### Gekaufte Hardware:
| Teil                                    | Spezifikation                          | Preis   |
|-----------------------------------------|----------------------------------------|---------|
| Raspberry Pi 5 4GB                      | Einzelboard                            | ~86€    |
| RPi Touch Display 2 (5")               | 720×1280, DSI, kapazitiv, Portrait     | ~43,50€ |
| TECNOIOT 2S BMS (5 Stk.)              | 10A Dauer, 20A Limit, Tiefentladeschutz| ~15€    |
| Acrylplatten 150×200×1mm (15 Stk.)    | Transparent, für Pepper's Ghost        | ~9€     |
| microSD 32GB+                           | A2, Class 10 (noch zu kaufen)          | ~10€    |
| **Gesamt**                              |                                        |**~164€**|

### Display-Entscheidung Begründung:
- DSI-Anschluss: GPIO bleibt frei für Adafruit Motor HAT
- 720×1280 Portrait nativ: Perfekt für 512×1024 Sprite, kein Rotation-Hack
- Display-Strom: Separater 5V-Anschluss auf eigener PCB (vom D36V50F5)
- Pepper's Ghost: LCD horizontal, Acryl 45°, schwarzer Hintergrund → Sprite "schwebt"

### Energiebudget:
- 2× 18650 (35A E-Zig-Zellen, ohne Schutz) → 2S BMS → Sicherung 10A → PCB
- D36V50F5: 5V/5.5A für RPi5 + Display (~2-3A normal)
- Motoren: 4× TT-Motor direkt @ 7.4V (~0.8A normal, ~6A Stall)
- Normalbetrieb gesamt: ~3A, Worst Case ~8.5A → Sicherung 10A passt

### Gehäuse:
- Eigenbau: Resin-Druck (12kg Material vorhanden)
- Design in Phase 4 (Inventor), nach Hardware-Prototyp-Test

## Architektur-Entscheidung: 3-Tier-System (2026-03-14)
- Tower = Hirn (LLM + TTS-Generierung, immer an)
- Laptop = Client (PC-Steuerung + Audio-Empfänger, wenn User auf Couch)

- TTS wird auf Tower generiert, WAV an aktiven Client gestreamt
- Laptop braucht AgentServer (FastAPI): /action/execute, /audio/play, /health

## Hardware-Entscheidung: Motor-Upgrade TT → JGB37-520 (2026-03-15)
- Grund: Gehäuse 1.138g + Elektronik ~760g + Chassis ~400g = ~2.3kg → TT-Motoren zu schwach
- Neuer Motor: JGB37-520 (12V, Hall-Encoder, 2-10 kg·cm je nach Ratio)
- Traglast mit 4× JGB37: 5-8kg → massig Reserve für ~3kg Roboter
- Encoder ermöglicht PID-Regelung auf Pico 2W
- Konsequenzen:
  - Neues Chassis nötig (JGB37 = Ø37mm Getriebe, deutlich größer als TT)
  - 4× Motor = 784g (statt 120g bei TT) → Gesamtgewicht steigt auf ~2.7-3kg
  - Motor-Driver: Adafruit Motor HAT evtl. zu schwach (1.2A/Kanal, JGB37 Stall = 2.8A)
    → Alternative evaluieren: L298N, BTS7960, oder Custom-Driver auf eigener PCB
  - Akku: 2S (7.4V) läuft, aber JGB37 ist für 12V spezifiziert (60% Drehzahl bei 7.4V)
    → 3S (11.1V) prüfen: BMS-Kompatibilität, D36V50F5 Eingangsbereich, PCB-Redesign
  - Mecanum-Räder: 60mm oder 80mm mit 6mm D-Shaft Aufnahme
- OFFEN: Akku 2S vs 3S → BMS und PCB prüfen
- OFFEN: Motor-Driver Wahl → Phase 4 PCB-Redesign
- OFFEN: Chassis-Design → nach Motor-Bestellung

## Hardware-Meilenstein: RPi5 Display + Avatar live (2026-03-17)

### Display am RPi5 in Betrieb
- RPi Touch Display 2 (5", 720×1280, Portrait) angeschlossen und funktional
- Problem: pygame (SDL 2.28.4) auf Bookworm Lite hatte keinen Display-Driver
  - Alle Driver fehlten: kmsdrm, fbcon, directfb, egl, wayland, x11 (nur dummy)
  - System-SDL2 war ohne Display-Backends kompiliert (normal für Lite)
- Lösung: `pip install pygame-ce` (Community Edition, bringt eigene SDL2 mit kmsdrm)
  - Drop-in Replacement, gleiche API, kein Code-Änderung
  - Start: `SDL_VIDEODRIVER=kmsdrm python scripts/start_rpi5.py`
- Saleria rendert auf physischem Display: 720×1280 fullscreen, 21 Komponenten, 30 FPS
- Steuerung über REST API funktioniert: POST /avatar {"emotion": "cheerful"} etc.

### Gehäuse-Design (Inventor CAD)
- Innenraum-Anordnung definiert: RPi5 zentral unten, Display horizontal darüber
- Pepper's Ghost: Display leuchtet nach oben → 45° Acrylscheibe → Betrachtungsfenster seitlich
- Display-Halterung: Nut zum Einlegen + 6 Niederhalter (3 pro Seite), schraubenfrei
- ULN2003 Treiberboard neben RPi5 platziert (kurze Kabelwege zum Stepper)
- Belüftung: Luftspalte unten (zwischen Wurzeln) → Kamineffekt → Abluft durch Ast rechts
  - RPi5 hat ~5cm Freiraum ringsum, passive Konvektion reicht für ~10W
- Innenraum wird matt-schwarz lackiert (Hologramm-Effekt)

### Drehteller-Konzept finalisiert
- Motor: 28BYJ-48 Stepper (5V, ULN2003 Treiber, 4096 Steps/Umdrehung)
  - Kein Servo: volle 360° möglich, leise, langsam, präzise
- Drehwinkel: ±180° (begrenzt durch Stromkabel)
- USB-C Verlängerung (150cm F→M) nach unten durch Drehachse verlegt
  - Einziges Kabel das die Grenze feststehend↔drehend überquert
  - DSI/CSI/Sensoren sitzen alle auf dem Drehteller, drehen sich mit
- Endanschlag: beidseitige Magnete als harter mechanischer Stopp
  - Hall-Sensor (A3144) erkennt gleichzeitig Home-Position
- ULN2003 VCC: extern 5V (nicht RPi5 GPIO, wegen 240mA Strombedarf)

### Bestellungen
- Bestellt (BerryBase): Raspberry Pi Camera Module 3 (75°), GY-BME280, APDS-9960
- Angekommen: 28BYJ-48 Stepper + ULN2003, A3144 Hall-Sensoren (20 Stück)
- Vorhanden: Dupont-Kabel-Set (F-F/M-M/F-M 10cm)

### Entscheidungen
- Keine PCB: Verdrahtung mit Dupont-Steckern + gedruckte Kabelführung reicht
- Kein interner Akku: USB-C Netzteil für Dauerbetrieb, Anker Powerbank für unterwegs
- Kamera normal (75°) statt Wide (120°): Schreibtisch-Abstand ~60-80cm, 75° erfasst genug
- BerryBase statt Amazon: kleineren Laden unterstützen, Kamera dort auch günstiger

- Phase "Intent-Classifier (Self-Learning Command-Routing)" aus PROJECT_ROADMAP.md entfernt
  - Problem existiert bei aktuellem Scope nicht, Regex+Keywords+LLM-Fallback reicht
  - Phasen umbenannt: Phase 10→9, Phase 11→10
- Konzept-Dokument erstellt: docs/concepts/phase-8-1-8-3-weather-timer-briefing.md
  - Phase 8.1 Wetter (Open-Meteo), 8.2 Timer & Erinnerungen, 8.3 Daily Briefing

### Entscheidungen
- **Pico 2 / Pico 2W → GESTRICHEN**: Alle Sensoren + Stepper direkt über RPi5 GPIO.

  Vorteil gegenüber Axiallager: radiale Stabilität auf 90mm Radius, ein Teil statt zwei.
- **Verbindungstechnik Drucksegmente**: Zapfen + Nut + Fixierstifte statt Schrauben.
  Weniger Gewindeeinsätze, besser für dünnwandiges Resin, formschlüssig.

- **APDS-9960** (Proximity/Gesture/Light): Am Sockel unten, Sensor-Fenster nach vorne (Richtung Nutzer).
  Board in Tasche, 0,25mm Luft ringsum. Nut als Schiene: reindrücken, Schwerkraft hält.
  Zwei 30°-Fasen links/rechts (Wandstärke 3mm vs 1mm asymmetrisch → Fasen öffnen IR-Kegel).
  Horizontal montiert — reicht für Proximity-Erkennung auf 60-80cm Schreibtischabstand.
- **BME280** (Temp/Humidity/Pressure): Am Boden, Sensor nach unten zeigend.
  Mit Schraube fixiert. NICHT im Luftauslass (würde RPi-Abwärme messen statt Raumtemperatur).

### Drehteller-Konstruktion (CAD in Arbeit)

  Motor dreht Welle → Zapfen fest → Motorgehäuse (+ Baum) dreht sich entgegengesetzt.
- Obere Platte: verbunden mit Innenring Drehkranz + Stepper-Gehäuse + Baumstamm
- Untere Platte: verbunden mit Außenring Drehkranz + Zentralzapfen (D-Aussparung für Motorwelle)
- Motorwelle: 0,3mm Spiel in D-Aussparung (nur Drehmoment, keine Radialkraft)

- Schwerpunkt: 3mm aus Mitte bei 98mm Lager-Radius → kein Kipp-Risiko
